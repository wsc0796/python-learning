# 07. 工作流进阶集成

> 前置：[[06-advanced-workflow-nodes]]
> 目标：消息卡片、多工作流串联、知识库集成、HTTP 插件调用外部 API
> 用时：约 12 分钟

---

## 一、消息卡片

### 用途

让你的 AI 回复不只是纯文本，而是结构化、可交互的卡片。

### 适用场景

- 飞书机器人（需要卡片消息格式）
- 豆包端展示（图文卡片）
- 办公自动化通知（任务卡片）
- 数据看板（指标卡片）

### 卡片结构

```json
{
  "title": "今日天气",
  "content": "北京 晴 26°C",
  "elements": [
    {"type": "text", "content": "适合户外活动"},
    {"type": "image", "url": "https://..."}
  ]
}
```

扣子有内置的卡片模板，配置回复节点时选择"卡片消息"类型即可。

## 二、多工作流串联

### 为什么需要串联

一个复杂任务拆成多个工作流，每个专注一件事：

```text
工作流 A：选题生成
  输出：10 个选题
    ↓
工作流 B：文案生成（循环调用 A 的每个选题）
  输出：10 篇文案
    ↓
工作流 C：配图生成（循环调用 B 的每篇文案）
  输出：10 条图文
    ↓
工作流 D：发布格式组装
  输出：可直接发布的完整内容
```

### 串联方式

1. **调用子工作流节点** — 在工作流中直接调用另一个工作流
2. **API 串联** — 工作流 A 生成结果 → HTTP 请求触发工作流 B

## 三、知识库 + 工作流

### 典型场景

```text
用户问："我们的退换货政策是什么？"
    ↓
工作流接收问题
    ↓
知识库节点搜索"退换货政策"
    ↓
LLM 节点：根据知识库搜索结果 + 用户问题 生成回答
    ↓
回复节点返回精准答案
```

这就是 **RAG（检索增强生成）** 的简化实现——先检索相关文档，再让 LLM 基于文档回答。

### 配置步骤

1. 在资源库中上传文档（PDF/Word/TXT/Markdown）
2. 等待索引完成并发布
3. 在工作流中添加"知识库"节点
4. 配置查询字段（通常是用户问题）
5. 将知识库返回的文本作为 LLM 节点的上下文

### 应用场景

| 场景 | 知识库内容 |
|------|-----------|
| 企业客服 | 产品手册、FAQ、合同条款 |
| 政策问答 | 法规文件、政策解读 |
| 内部助手 | 技术文档、Wiki、会议纪要 |
| 教育助手 | 教材、讲义、习题答案 |

## 四、HTTP 插件集成

### 用途

调用你自己的后端 API，或者任何第三方 REST API。

### 配置

```text
请求方式: GET / POST
URL: https://your-api.com/endpoint
Headers: {"Authorization": "Bearer xxx"}
Body: {"param1": "{{开始节点.xxx}}"}
```

### 实战场景

```text
工作流 → HTTP 插件 → 你的 FastAPI 后端
                         ↓
                    查数据库、处理业务逻辑
                         ↓
                    返回结果给工作流
                         ↓
                工作流用 LLM 节点把结果转成自然语言回复
```

### 和你技术栈的关系

```
扣子工作流（前端/编排层）
    ↓ HTTP 插件
你的 FastAPI 后端（业务逻辑层）
    ↓
数据库 / 其他服务（数据层）
```

**扣子不替代你的后端——它在你的后端之上做了一层"AI 编排"。**

## 五、实际架构模式

```
用户 → 扣子智能体（对话界面）
         ├── 简单问答 → 直接回复
         └── 复杂任务 → 触发工作流
                          ├── LLM 生成内容
                          ├── 知识库检索
                          ├── HTTP 调 FastAPI 后端
                          └── 回复（文本/卡片/图片）
```

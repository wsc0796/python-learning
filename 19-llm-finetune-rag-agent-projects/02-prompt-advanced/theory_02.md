读完约 12 分钟。

## 一、In-context Learning（情境学习）

不更新模型参数，只在 Prompt 中提供示例，让模型根据上下文完成任务。

### Zero-shot（零样本）

```text
请判断情感倾向：
句子：这款手机续航很差。
情感：
```

模型直接回答，不给任何例子。

### Few-shot（少样本）

```text
句子：这家店服务很好。
情感：正面

句子：物流太慢了。
情感：负面

句子：这个耳机音质不错。
情感：
```

给了两个示例，模型就能举一反三。

### 关键理解

```
Zero-shot：靠模型预训练时的知识
Few-shot：靠示例让模型理解任务格式

两者都不改变模型参数。
两者都属于 In-context Learning。
```

## 二、Instruction（指令）

Instruction 是对任务的明确说明。好的 Instruction 包含：

| 要素 | 说明 | 示例 |
|------|------|------|
| 角色 | 模型扮演什么角色 | "你是一名专业的信息抽取助手" |
| 任务 | 要做什么 | "从文本中抽取商品名称和情感" |
| 输入 | 输入是什么 | "用户评论" |
| 输出格式 | 输出的结构 | "JSON 格式" |
| 约束 | 不能做什么 | "不要猜测，只抽取原文中出现的信息" |
| 示例 | 输入输出对 | "输入：... 输出：..." |

### Instruction 模板示例

```python
instruction = """你是一名专业的物流客服助手。

任务：根据用户咨询内容，判断问题类型。

可选类型：
- 时效问题（快递超时未到）
- 丢件问题（快递丢失）
- 破损问题（商品损坏）
- 地址问题（地址错误无法派送）

输出格式：只输出类型名称，不要多余文字。

用户：{user_input}
类型："""
```

## 三、CoT：Chain of Thought（思维链）

核心思想：让模型先写出推理过程，再给出答案。

### 为什么要 CoT

大模型直接回答复杂问题时容易出错。
让模型"一步步思考"能大幅提高准确率。

### Zero-shot CoT

```text
问题：小明有 3 个苹果，又买了 5 个，送给朋友 2 个，还剩几个？

请一步步思考。
```

输出：
```text
小明原来有 3 个苹果。
又买了 5 个，所以共有 3 + 5 = 8 个。
送给朋友 2 个，所以还剩 8 - 2 = 6 个。
答案：6 个。
```

### Few-shot CoT

给几个带推理过程的示例，让模型学会"先推理再回答"的模式。

### CoT 适用场景

| 适合 | 不适合 |
|------|--------|
| 数学题 | 简单事实问答 |
| 逻辑推理 | 情感分类 |
| 多步骤规划 | 关键词提取 |
| 复杂判断 | 翻译任务 |
| 代码分析 | 模板填充 |

---

## 四、CoT 的常见变体

| 方法 | 核心思想 |
|------|---------|
| Zero-shot CoT | 加一句"请一步步思考" |
| Few-shot CoT | 给带推理过程的示例 |
| Self-Consistency | 多次推理，投票选最一致的答案 |
| ReAct | 推理 + 行动循环（适🈴 Agent） |

## 五、提示词工程 vs 微调的关键认知

```
提示词工程 = 在输入上做文章，不改变模型
    优点：零成本、快速迭代
    缺点：受限于模型能力天花板

微调 = 在模型参数上做文章，改变模型
    优点：可以突破模型原有能力
    缺点：需要数据、算力、时间

实际项目中：先试提示词工程，不够再考虑微调。
```

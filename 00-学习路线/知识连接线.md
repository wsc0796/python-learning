## 核心连接线（按调用链走）

以下按照 **一个 HTTP 请求的完整旅程** 串起所有核心模块：

### 站台 1：请求到达 → `25-function-basics` → `11-closure-decorator`

```python
@app.post("/tasks")          # ← 装饰器语法，来自 11-closure-decorator
def create_task(...):        # ← 函数定义，来自 25-function-basics
```

**连接逻辑**：`@app.post("/tasks")` 是"带参数的装饰器"——`app.post("/tasks")` 先返回一个装饰器函数，再把 `create_task` 传进去注册成路由。不理解闭包就看不懂这行代码在干什么。

| 来自 | 通向 | 连接点 |
|------|------|--------|
| `25-function-basics` | `11-closure-decorator` | 函数是一等公民 → 函数可以当参数传 → 装饰器就是这个思想的语法糖 |
| `11-closure-decorator` | `10-fastapi/main.py` | `@app.get()` / `@app.post()` 的本质就是闭包+装饰器 |

---

### 站台 2：请求体校验 → `07-pydantic` → `06-dict` → `01-fstring`

```python
task: TaskCreate           # ← Pydantic BaseModel，来自 07-pydantic
```

**连接逻辑**：

```text
01-fstring (格式化字符串)
  → 06-dict (字典是 JSON 的 Python 形态)
    → 07-pydantic (BaseModel = 有校验规则的 dict)
      → models.py 里的 TaskCreate / TaskRead
```

Pydantic 本质上做了两件事：(1) 把 JSON 转成 Python 对象 (2) 按你声明的规则校验每个字段。`01-fstring` 学会的字符串操作在 Pydantic 的 `Field(min_length=1)` 错误消息里天天出现。

| 来自 | 通向 | 连接点 |
|------|------|--------|
| `01-fstring` | `07-pydantic` | `Field(min_length=1)` 校验失败时，pydantic 生成的错误消息就是格式化的字符串 |
| `06-dict` | `07-pydantic` | `model_dump()` → dict, `model_validate(dict)` → BaseModel |
| `07-pydantic` | `10-fastapi/models.py` | TaskCreate / TaskRead 的定义 |

---

### 站台 3：依赖注入 → `08-typing-di` → `05-module-types`

```python
TaskServiceDep = Annotated[TaskService, Depends(get_task_service)]
```

**连接逻辑**：

```text
05-module-types (import/模块组织)
  → 08-typing-di (类型标注 + 依赖注入)
    → dependencies.py (把创建对象的代码集中管理)
```

`Depends(get_task_service)` 的意思是："FastAPI，这个参数你不用管，调用 `get_task_service()` 就能拿到。"这跟 `08-typing-di` 里学的 `ReportService(printer: Printer)` 完全一样的思想——**调用者不关心依赖怎么创建，只声明它需要什么**。

| 来自 | 通向 | 连接点 |
|------|------|--------|
| `05-module-types` | `08-typing-di` | `Protocol` / `ABC` 定义行为约定，跨模块共享接口 |
| `08-typing-di` | `10-fastapi/dependencies.py` | `Depends()` = Python DI 的 FastAPI 版本 |
| `08-typing-di` | `10-fastapi/repository.py` | `TaskRepository(Protocol)` = 行为约定，不绑定具体实现 |

---

### 站台 4：业务逻辑 → `09-crud` → `06-dict` → `03-file-exception`

```python
# service.py
def create_task(self, task_create: TaskCreate) -> TaskRead:
    tasks = self.repository.list_tasks()       # ← 读数据
    next_id = max(...) + 1                      # ← 业务逻辑
    task = TaskRead(id=next_id, ...)            # ← 组装对象
    self.repository.save_tasks(tasks)           # ← 写数据
```

**连接逻辑**：

```text
09-crud (增删改查的通用模式)
  → 06-dict (内存字典存储)
    → 03-file-exception (JSON 文件读写 + try/except)
      → repository.py (JsonTaskRepository)
```

`09-crud` 教会你 CRUD 的四步模式（校验→操作→保存→返回），`service.py` 就把这个模式应用到具体业务。`repository.py` 里的 `json.load()` / `json.dump()` 是 `03-file-exception` 的直接应用。

| 来自 | 通向 | 连接点 |
|------|------|--------|
| `09-crud` | `10-fastapi/service.py` | Create/Read/Update/Delete 的 Python 实现 |
| `06-dict` | `10-fastapi/service.py` | `list_tasks()` 返回 `list[TaskRead]`，本质是列表+字典的组合操作 |
| `03-file-exception` | `10-fastapi/repository.py` | `json.load(file)` / `json.dump(...)` / `FileNotFoundError` |

---

### 站台 5：错误处理 → `03-file-exception` → HTTP 状态码

```python
# main.py
try:
    return service.get_task(task_id)
except TaskNotFoundError as exc:
    raise HTTPException(status_code=404, detail=str(exc)) from exc
```

**连接逻辑**：

```text
03-file-exception (try/except/raise)
  → 09-crud (自定义异常 TaskNotFoundError)
    → 10-fastapi/main.py (业务异常 → HTTPException)
```

关键边界感：**service.py 只抛 `TaskNotFoundError`，不提 HTTP；main.py 负责把异常翻译成 404。** 这是 `09-crud` 阶段就建立的"业务层不关心传输层"原则。

| 来自 | 通向 | 连接点 |
|------|------|--------|
| `03-file-exception` | `10-fastapi/service.py` | `raise TaskNotFoundError(task_id)` |
| `09-crud` | `10-fastapi/main.py` | `except TaskNotFoundError → HTTPException(404)` |
| `10-fastapi/main.py` | 浏览器/前端 | 422 (Pydantic校验失败) vs 404 (业务找不到) |

---

### 站台 6：响应输出 → `07-pydantic` → `01-fstring`

```python
@app.post("/tasks", response_model=TaskRead, status_code=201)
```

**连接逻辑**：FastAPI 拿到 `TaskRead` 对象后，调用 `model_dump()` 转成 dict，再序列化为 JSON 返回。这一整条链从 `01-fstring`（字符串格式化）→ `06-dict`（字典操作）→ `07-pydantic`（模型序列化）穿过来。

## 进阶连接线（Layer 2 → Layer 4）

### 装饰器 → Agent Tool 注册

```text
11-closure-decorator (@app.get)
  → 理解"把函数注册到某个系统"
    → 20-agent-architecture-harness (@tool 装饰器)
```

LangChain/LangGraph 里的 `@tool` 装饰器和 FastAPI 的 `@app.get` 是同一个模式：**用一个装饰器把普通函数注册到框架中**。

### 异步 → LLM API 调用

```text
17-context-iterator-async (async/await)
  → 10-fastapi (async def 路由)
    → 19-llm-finetune-rag-agent (async 调用 LLM API)
```

FastAPI 的 `async def` 和调用 OpenAI/DeepSeek API 时的 `await` 是同一条 `asyncio` 事件循环。学会了 `17-context-iterator-async` 里的 `async for`，就能看懂 Agent 里流式输出的代码。

### Pydantic → LLM Function Calling

```text
07-pydantic (BaseModel, Field)
  → OpenAI Function Calling 的 parameters 定义
    → 19-llm-finetune-rag-agent (structured output)
```

当你用 `response_model=TaskRead` 声明 FastAPI 输出格式时，和用 Pydantic 定义 LLM 的 function call schema 是完全一样的思路——**先定义数据结构，框架自动校验和序列化**。

### DI → Agent 依赖管理

```text
08-typing-di (依赖注入)
  → 10-fastapi (Depends)
    → 20-agent-architecture-harness (Agent 的工具/模型/记忆注入)
```

Agent 需要 LLM、工具列表、记忆存储——这些不应该在 Agent 内部 `import` 写死，而应该从外部注入。这和 `TaskService(repository=JsonTaskRepository(...))` 的思想一致。

## 今天的聚焦点 (2026-06-13)

你今天要串的核心链路只有一条：

```
HTTP POST /tasks
  → @app.post("/tasks")           [11-closure-decorator]
    → TaskCreate 校验请求体        [07-pydantic]
      → Depends(get_task_service) [08-typing-di]
        → service.create_task()   [09-crud]
          → repository.list_tasks()    [03-file-exception]
          → max(id) + 1 生成新ID      [06-dict + 21-list-deep]
          → repository.save_tasks()   [03-file-exception]
        → return TaskRead          [07-pydantic model_dump]
  → HTTP 201 + JSON 响应
```

这条链上的每一个括号就是你和之前知识的连接点。如果某个括号看不懂，追溯回去补那个模块的 `theory_*.md` 或 `practice_*.py`。

## 使用方式

1. **学新模块前**：查快速查线表，确认前置依赖是否扎实
2. **卡住时**：沿着连接线往回追溯，找到断点
3. **复盘时**：对照总链路图，画出今天覆盖了哪几个站台
4. **面试准备时**：用总链路图做"从HTTP请求到数据库"的完整复述

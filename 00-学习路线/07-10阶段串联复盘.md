# 07-10 阶段串联复盘

## 这几章真正串起来的主线

装饰器不是单独学完就结束，它是理解 FastAPI 路由注册的入口：

```python
@app.get("/tasks")
def list_tasks():
    ...
```

这段代码本质上就是“带参数的装饰器”：`app.get("/tasks")` 先返回一个装饰器，再把下面的函数注册成路由。

## 07 Pydantic：数据入口要先校验

Pydantic 在这一段里承担 DTO 和参数校验的角色：

- `BaseModel`：声明数据结构
- `Field`：声明约束，例如 `min_length`、`gt`、`ge`、`le`
- `model_dump()`：模型转字典，适合存入内存或 JSON
- `model_validate()`：从字典重新校验成模型
- `model_copy(update={...})`：在不直接乱改对象的情况下生成更新后的模型

要记住：类型提示本身不负责运行时校验，Pydantic 才负责把输入数据真正校验一遍。

## 08 类型提示 + DI：业务代码不要写死依赖

这一章的重点不是“类型写得漂亮”，而是让代码可替换：

```python
class ReportService:
    def __init__(self, printer: Printer) -> None:
        self.printer = printer
```

`ReportService` 不关心你传进来的是控制台打印、文件打印，还是测试用的假对象。只要它有 `print_doc()` 行为，就能用。

这就是后面 FastAPI `Depends()` 的思想基础：函数需要什么依赖，由外部注入，而不是在函数里面写死。

## 09 CRUD：把数据模型变成业务动作

CRUD 是把前面的知识串起来：

- Create：先用 Pydantic 校验，再存入 `task_db`
- Read：用类型清晰的函数返回列表或单个任务
- Update：修改完成状态
- Delete：删除成功返回布尔值，或用自定义异常表达失败
- Save/Load：用 JSON 做最小持久化

这里最重要的边界感是：纯 Python CRUD 层只处理业务和数据，不要掺 HTTP 状态码。

## 10 FastAPI：把 CRUD 暴露成 HTTP 接口

FastAPI 负责把外部 HTTP 请求接进来：

- 路径参数：`/tasks/{task_id}`
- 查询参数：`/tasks?completed=true`
- 请求体：`TaskCreate`
- 响应模型：`response_model=TaskRead`
- 错误转换：业务里找不到任务，API 层返回 404
- 依赖注入：`Depends(get_task_service)`

一个比较清晰的心智模型：

```text
HTTP 请求
  -> FastAPI 路由
  -> Pydantic 校验请求体/参数
  -> Depends 注入服务
  -> CRUD/Service 执行业务
  -> Pydantic 响应模型输出 JSON
```

## 当前阶段还需要注意的点

1. 输入模型和输出模型最好分开，例如 `TaskCreate` 不包含 `id`，`TaskRead` 包含 `id` 和 `completed`。
2. 纯 Python 文件里可以抛自定义异常；FastAPI 路由里再把它转换成 `HTTPException`。
3. 内存字典只适合练习；真正项目下一步要换成数据库，这时 DI 的价值就会更明显。
4. Pydantic v2 推荐使用 `model_dump()`、`model_validate()`、`model_copy()`，不要再按 v1 的 `.dict()` 作为主写法。
5. FastAPI 路由函数被 `@app.get()`、`@app.post()` 装饰后，不是普通脚本入口，所以不要在模块导入时直接执行打印或测试逻辑。

## 二、真正卡住的点

### 卡点 1：不知道怎么"凭空"写出代码

**表现：** 知道需求是什么（比如"保存笔记"），但不知道代码怎么从零开始。

**突破：** 建立了"需求→人话步骤→每步对应一行代码"的翻译工作流：

```
需求：保存 notes
  → 步骤1：文件夹要存在 → mkdir()
  → 步骤2：对象不能直接写 JSON → model_dump()
  → 步骤3：多个对象要循环转 → [... for note in notes]
  → 步骤4：包成 {"notes": [...]} → payload = {...}
  → 步骤5：打开文件 → with open(...)
  → 步骤6：写进去 → json.dump(...)
```

### 卡点 2：对照模板写代码时漏翻译

**表现：** 写 `from repository import JsonTaskRepository` 而不是 `JsonNoteRepository`；写 `return task_service` 而不是 `return note_service`。

**原因：** 识别强、回忆弱——看模板时知道是什么，自己写时脑子里还残存着旧名字。

**突破：** 每次写完对着翻译表逐行检查。左边原文件 + 右边新文件 = 对照翻译，不是默写。

### 卡点 3：`self.repository.xxx()` 看不懂点号链

**表现：** 不理解 `self.repository.list_notes()` 里的 `.` 是什么。

**突破：** `.` = 往里走一层，`()` = 执行。遥控器→电池仓→取电池，每一层只做一件事。

### 卡点 4：依赖注入到底是什么

**表现：** 虽然看了笔记，但无法用自己的话解释。

**突破：** DI = `self.x = 参数` 而不是 `self.x = Xxx()`。放弃创建权，换来可替换性。用充电头类比：手机不焊死充电器，不焊死就能换。

## 七、方法论收获

**写代码不是凭空创造，是翻译。** 需求→人话步骤→每步对应语法。对照已有项目时，左边开原文件右边写新文件，一行一行翻译，逻辑照抄只改名字。这方法适用于所有框架和语言。

**"看懂"和"会写"是两道坎。** 看懂是识别（recognition），会写是回忆（recall）。它们用大脑不同的回路。必须通过不看模板写一遍来打通。偷看可以，但看完要关掉继续写——逼的是回忆能力。
